# Persistence

Rules for ORM design, query placement, session scope, and migrations.

Load when: defining or modifying models, writing queries, handling sessions, or changing schema.

## Rules

### PER-1 Keep ORM models focused on mapping [MUST]

`models/` should define tables, columns, nullability, keys, and indexes. Do not put API shapes, business workflows, or session management there. Schema roles: **VAL-2** (`VALIDATION.md`).

### PER-2 Make required columns explicit [MUST]

If a column is required, declare `nullable=False`. Do not rely on defaults that silently allow bad data.

### PER-3 Index real filter and join paths [SHOULD]

Columns used often in `WHERE`, `ORDER BY`, or `JOIN` clauses should usually be indexed deliberately.

### PER-4 Open sessions at the edge and pass them inward [MUST]

Session creation belongs at the workflow edge — routes, CLI commands, or top-level scripts. CRUD and service functions receive `db: Session` explicitly; they do not create hidden sessions internally. FastAPI routes: **PY-13** (`PYTHON.md`).

### PER-5 Keep commit ownership explicit [MUST]

Read paths must not commit. Multi-step writes should share one clear transaction scope.

### PER-6 Reusable queries belong in CRUD [SHOULD]

If a query is used by more than one caller, give it a named home in `crud/` instead of duplicating it in routes, scripts, or notebooks. See PER-11 for how CRUD classes are structured.

### PER-7 Every schema change needs a migration [MUST]

No schema change is complete without a migration that describes the change clearly and can be reviewed independently.

### PER-8 Review autogenerated migrations before commit [MUST]

Autogenerated migrations are drafts, not truth. Check nullability, accidental drops, column types, indexes, ordering, and rollback logic before commit.

### PER-9 Load relationships explicitly [SHOULD]

Default to explicit eager loading or explicit access patterns when relationships matter. Do not rely on hidden lazy loading across unclear session boundaries.

### PER-10 Holy trinity when adding a table [MUST]

Adding a table requires three coordinated changes; missing any one is a silent bug:

1. Define the model in `models/<table>.py` using SQLAlchemy 2.0 (`Mapped[...]`, `mapped_column(...)`).
2. Import it in `models/__init__.py` and add it to `__all__` — Alembic only sees imported models.
3. Generate and review the migration: `uv run alembic revision --autogenerate -m "..."`, then `uv run alembic upgrade head`.

### PER-11 CRUD inherits from CRUDBase with three generics [MUST]

Reusable CRUD classes inherit `CRUDBase[Model, PydanticSchema, PanderaDFSchema]` and are instantiated at module scope:

```python
class CRUDHeadline(CRUDBase[Headline, HeadlineSchema, HeadlineDFSchema]):
    pass

headline = CRUDHeadline(Headline, HeadlineDFSchema)
```

Export the instance from `crud/__init__.py` so callers use `crud.headline`, not the class.

Method names should express behavior: prefer `get_by_source`, `get_latest`, `count_by_day`, `upsert_batch` over vague names like `fetch`, `load`, or `query`.

### PER-12 SQLite is for local only [SHOULD]

SQLite is acceptable for local development and unit tests. Behavior tied to a different production dialect (Postgres) must be verified against that dialect — see **TEST-4** (`TESTING.md`).

## Anti-patterns

- Business logic inside ORM models because "the data is already there".
- Hidden session creation inside helpers.
- Committing inside read functions.
- Duplicating the same query in several routes or scripts.
- Editing an already-applied migration instead of creating a new one.
- Building outputs from ORM `.__dict__` instead of selecting fields deliberately.
